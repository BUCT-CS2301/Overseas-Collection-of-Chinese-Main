# 1. 项目概述

知识图谱构建子系统是"海外藏中国文物知识管理与服务平台"的核心数据基座，负责从海外博物馆网站爬取中国文物信息，经数据清洗、本体建模、三元组转换、实体对齐与去重后，存储至图数据库与关系型数据库的双层架构中，为知识服务子系统、知识问答子系统、掌上博物馆和后台管理子系统提供结构化的文物知识数据支撑。

本项目管理计划用于明确 5 周开发周期内知识图谱构建子系统的范围、分工、进度、质量要求、风险应对和交付物，确保各成员围绕统一的数据格式规范与本体模型协同开发，并能在课程验收前完成可运行、可查询、可对接的知识图谱数据产品。

# 2. 项目目标与成功标准

- 完成至少 3 家指定海外博物馆（克利夫兰艺术博物馆、纳尔逊-阿特金斯艺术博物馆、宾夕法尼亚大学考古与人类学博物馆）的中国文物数据爬取，有效记录不少于 5000 件，图片下载成功率不低于 90%。

- 完成数据清洗与质量控制，统一年代格式、文物类型分类，去除重复记录，验证图片链接有效性，输出 UTF-8 编码的标准化 CSV 文件。

- 参照 CIDOC-CRM 文化遗产本体标准，完成本体建模，定义文物、博物馆、朝代、艺术家、地点等实体类型及收藏于、创作于、属于朝代、材质为、类型为等关系类型。

- 完成三元组转换，将清洗后的结构化数据映射为（主体-谓词-客体）形式，使用 RDFLib 或 RMLMapper 生成 RDF 三元组数据。

- 完成实体对齐与去重，对跨数据源的艺术家、朝代、地点等共享实体进行识别与合并，建立实体唯一标识符（URI）。

- 完成数据补充，从百度百科、维基百科等外部来源定向补充书画作者生平、朝代背景、文物介绍等缺失信息，并标注来源与补充日期。

- 完成双层存储架构搭建，图数据库（Neo4j）存储全部三元组并支持 Cypher 查询与关系遍历，关系型数据库（MySQL）存储文物详细数据与系统业务数据。

- 支持增量爬取与数据更新，记录每次爬取的时间、数量与变更详情，对更新实体自动触发三元组重建与图数据库同步。

- 提交完整的字段说明文档、数据质量自查报告和爬取说明文档，数据可通过课程验收标准中的必填字段完整率、图片有效性、编码正确性和 URL 有效性检查。

# 3. 项目范围

## 3.1 必做范围

- 数据爬取：爬取克利夫兰艺术博物馆、纳尔逊-阿特金斯艺术博物馆和宾夕法尼亚大学考古与人类学博物馆的全部中国文物信息，包括文物名称、图片（原图）、年代、类型、材质、介绍文本、详情页 URL 等字段，按附录格式保存为 UTF-8 编码的 CSV 文件。

- 数据清洗与质量控制：字段标准化（统一年代格式、文物类型分类）、去重处理、图片有效性验证、空值处理。

- 数据建模：参照 CIDOC-CRM 标准设计本体模型，定义至少 5 类实体（Artifact、Museum、Dynasty、Artist、Location）和至少 5 类关系（收藏于、创作于、属于朝代、材质为、类型为）。

- 三元组转换：使用 RDFLib 或 RMLMapper 将 CSV 数据批量转化为 RDF 三元组。

- 实体对齐与去重：跨数据源共享实体识别与合并，建立 URI，保留原始来源信息。

- 数据补充：从百度百科、维基百科爬取补充信息，建立与原始数据的关联。

- 数据存储：Neo4j 图数据库存储三元组，MySQL 关系型数据库存储结构化数据。

- 数据更新与增量爬取：支持增量爬取，记录爬取日志，自动触发三元组重建与同步。

- 文档交付：字段说明文档、爬取说明文档、数据质量自查报告。

## 3.2 可选增强范围

- 将三元组发布为链接开放数据（LOD），支持 SPARQL 端点查询。

- 使用 Virtuoso 替代或并行 Neo4j 作为图数据库，支持 SPARQL 检索。

- 使用 Protégé 进行本体可视化设计与一致性验证，导出 OWL 文件。

- 建立自动化数据质量监控脚本，定期检测字段完整率与图片有效性。

- 实现多语言实体对齐，支持中英文实体名称的自动匹配。

- 构建数据血缘追踪体系，记录每条三元组从原始数据到最终存储的完整转换链路。

## 3.3 不在本阶段范围

- 不负责知识服务子系统的 Web 前端开发（浏览、查询、可视化页面）。

- 不负责知识问答子系统的大语言模型问答能力开发。

- 不负责掌上博物馆的 HarmonyOS App 开发。

- 不负责后台管理子系统的 Web 管理端开发。

- 不负责非指定博物馆的数据爬取（课程要求之外的博物馆不在本次范围内）。

# 4. 技术路线与系统架构

数据爬取采用 Python Scrapy 框架为主，针对 JavaScript 动态渲染页面辅以 Selenium / Playwright；数据清洗使用 Pandas 进行字段标准化、去重和空值处理；本体建模参照 CIDOC-CRM 标准，使用 Protégé 进行可视化设计或手动定义 OWL/RDF Schema；三元组转换采用 RDFLib（编程方式）或 RMLMapper（映射规则方式）将 CSV 数据批量转化为 RDF 三元组；实体对齐基于字符串相似度与规则匹配实现跨源实体合并；数据存储采用 Neo4j（图数据库，存储三元组，支持 Cypher 查询与关系遍历）+ MySQL（关系型数据库，存储结构化业务数据）的双层架构；增量更新通过爬取时间戳与变更日志实现增量爬取与自动同步。

| **层级** | **主要内容** | **关键输出** |
|----|----|----|
| 数据采集层 | 博物馆网站爬取、反爬处理、图片下载、增量爬取。 | CSV 数据文件、图片资源、爬取日志 |
| 数据处理层 | 字段标准化、去重、空值处理、数据翻译、数据补充。 | 清洗后 CSV、补充数据、翻译数据 |
| 知识建模层 | CIDOC-CRM 本体设计、实体与关系定义、三元组转换、实体对齐与去重。 | 本体模型（OWL/RDF Schema）、三元组数据（RDF/Turtle）、URI 映射表 |
| 数据存储层 | Neo4j 图数据库、MySQL 关系型数据库、数据同步与增量更新。 | 图数据库实例、关系型数据库实例、同步脚本 |
| 接口服务层 | 对外提供文物列表、详情、筛选字典、图谱关系、统计指标等数据接口。 | 接口文档、API 端点、数据导出 |

# 8. 质量管理计划

- 需求质量：核心爬取字段、本体实体与关系类型必须能对应课程说明中的数据格式要求和 CIDOC-CRM 建模标准。

- 数据质量：核心必填字段（object_id、title、period、type、description、museum、detail_url、image_url、crawl_date）不得有空值；图片须为原图而非缩略图；CSV 文件统一使用 UTF-8 编码，确保中文字符无乱码。

- 清洗质量：年代格式统一（如"Tang Dynasty"或"618–907 AD"），文物类型分类一致，重复记录合并率需达 95% 以上，图片链接有效性验证通过率不低于 90%。

- 建模质量：本体模型需覆盖课程要求的至少 5 类实体和 5 类关系，实体与关系定义需有明确语义说明，避免歧义。

- 三元组质量：每条三元组需通过格式验证（RDF 语法正确），URI 命名遵循统一规范，谓词需在本体模型中有明确定义。

- 对齐质量：跨数据源共享实体（艺术家、朝代、地点）对准确率不低于 90%，对齐结果需保留原始来源信息，支持追溯。

- 接口质量：数据接口返回的 JSON 结构需与接口文档一致，异常情况（查询无结果、参数错误）需返回明确错误码和提示信息。

- 性能质量：图数据库查询响应时间不超过 3 秒（常规关系遍历），关系型数据库查询响应时间不超过 1 秒（单表查询）。

# 1. 项目概述

海外文物知识服务子系统是平台面向普通用户、研究者和课程验收人员的 Web 端服务入口，主要基于知识图谱构建子系统提供的文物数据，为用户提供文物浏览、检索查询、图谱与统计可视化、用户登录注册和个人中心等功能。系统界面设计参考国际一流博物馆网站的浏览与检索体验，强调数据可读性、交互清晰度和服务稳定性。

本项目管理计划用于明确 5 周开发周期内的范围、分工、进度、质量要求、风险应对和交付物，确保各成员围绕统一前端框架与接口规范协同开发，并能在课程验收前完成可运行、可演示、可部署的知识服务子系统。

# 2. 项目目标与成功标准

- 完成 Web 端基础框架、首页、路由、用户登录注册和用户中心，为各功能模块提供统一入口。

- 实现数据浏览功能，支持列表/卡片视图切换、多维筛选、排序、文物详情、相关文物推荐和文物对比。

- 实现数据查询功能，支持关键字检索、高级组合查询、知识图谱关系查询、分页、二次筛选和查询结果导出。

- 实现数据可视化功能，包含知识图谱关系图、文物时间轴、地理分布图和统计分析看板。

- 完成服务器部署，保证前端页面、接口配置、静态资源、跨域配置和演示环境稳定可用。

- 系统在主流桌面浏览器中可正常访问，核心页面无明显布局错乱，主要接口异常时有明确提示。

## 知识问答子系统 
- 项目管理计划

- 知识问答子系统 文档版本：v1.0编制人：薛维浩 编制日期：2026.5.16
### 一、项目概述
1.1 项目背景
本项目为软件工程综合实验课程设计，是 “海外藏中国文物知识管理与服务平台” 的核心子系统之一。本子系统聚焦文物知识问答场景，基于知识图谱与大语言模型构建检索增强生成（RAG）架构的问答服务，为用户提供精准、可溯源的文物知识自然语言问答能力，并与掌上博物馆等子系统完成服务集成。

1.2 子系统目标
基于 “知识图谱 + 大语言模型” 的 RAG 架构，开发文物知识问答服务，实现以下核心功能：
基础问答：覆盖 10 类以上单实体单属性的简单文物知识问答，无数据时明确告知用户；
答案溯源：每条回答标注数据来源、提供原始数据链接，区分事实性内容与大模型补充描述；
拓展能力（选做）：多轮对话、复杂推理问答、问答质量反馈、文物文化相关闲聊与导览；
服务集成：提供标准化接口，支持掌上博物馆等子系统调用问答能力。
通过本子系统，交付高可信度、交互友好的文物知识问答服务，解决大模型幻觉问题，为用户提供专业、精准的文物知识解答。

## 1. 项目背景与目标

### 1.1 项目背景
本项目为北京化工大学“软件工程综合实验”课程设计，“海外藏中国文物知识管理与服务平台”的移动端组成部分。本子系统面向手机用户，提供沉浸式的文物浏览、语音导览、以图搜图、用户互动等功能，与知识服务、知识问答、后台管理等子系统协同，形成完整的文物知识服务生态。

### 1.2 子系统目标
基于 HarmonyOS + ArkTS 开发一款功能完整、体验流畅的 App，实现：
- **文物浏览**：首页瀑布流/卡片视图、多维筛选排序、详情页（高清图片、属性、相关推荐）、文物对比。
- **语音导览**：文物语音讲解、语音搜索、语音问答（对接知识问答子系统）。
- **以图搜图**：拍照/相册选取，返回相似文物及相似度。
- **用户交互**：登录注册、点赞收藏、评论留言、上传照片（审核后显示）。
- **个人信息**：个人主页、浏览记录、收藏夹、上传内容管理。

成功标准：通过课程验收，App 可独立运行，并与其他子系统成功集成。

## 5. 质量保证与测试策略

### 5.1 质量控制点
- **代码规范**：统一使用 `const`/`let`，禁止 `any`；组件命名 `PascalCase`；每次提交前运行 `Clean Project` 确保无编译错误。
- **UI/UX 一致性**：边距、颜色、字体大小统一使用常量；所有页面包含加载状态、空状态、错误状态。
- **数据可靠性**：Mock 数据与真实接口数据结构保持一致；字段缺失时显示“暂无信息”。
- **性能要求**：列表图片懒加载；详情页大图支持缩放。

### 5.2 测试计划
| 测试类型 | 覆盖功能 | 执行方式 | 通过标准 |
|----------|----------|----------|----------|
| 单元测试 | 工具类（PreferencesUtil、HttpApi） | 模拟调用，验证返回值 | 核心方法正确率 100% |
| 功能测试 | 所有必做功能（按任务分配文档） | 手工测试 + 测试用例记录 | 无阻断性错误，功能符合需求 |
| 兼容性测试 | 模拟器 + 真机 | 运行全部核心流程 | 布局无错位，交互正常 |
| 集成测试 | 调用真实后端接口 | 逐个页面验证数据正确性 | 所有接口返回数据正常展示 |
| 回归测试 | 每次合并 main 前 | 运行核心流程（登录、浏览、搜索） | 原有功能不被破坏 |

### 5.3 缺陷管理

| **后端**     | Spring Boot 3.x + Java 17                | 生态成熟，事务支持良好，适合权限、日志、备份等企业级功能。   |
| **安全认证** | Spring Security + JWT (或 Sa-Token)      | 实现RBAC、动态权限、操作日志记录                             |
| **数据库**   | MySQL 8.0 + Redis 7.x                    | MySQL存业务数据，Redis做缓存（权限、会话、实时统计）         |
| **图数据库** | Neo4j (引用，非独立建设)                 | 后台通过连接共享的图数据库进行数据查看、编辑与一致性检查。   |
| **对象存储** | 阿里云OSS                                | 存储用户上传的图片、视频及数据库备份文件。                   |
| **任务调度** | Quartz / XXL-JOB 或 Spring @Scheduled    | 定时备份、数据统计、自动审核任务                             |
| **日志监控** | Logback + MySQL                          | 结构化存储操作/安全日志                                      |
| **部署运维** | Docker + Docker Compose                  | 实现前后端一键部署，统一测试环境。                           |

> **架构说明**：子系统复用平台统一的用户体系，用户表可共享但虚拟划分“前台用户”与“后台管理员”，两者登录逻辑及Token生成策略在后端通过标识位字段严格分离。
